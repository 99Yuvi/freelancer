Authentication & Accounts
6 test cases
AUTH-01 Register as Client ▼
ℹ️ Do this in Browser A. You'll use this as the hiring client throughout all tests.
Steps
  1. Open http://localhost:5173 → click Get started
  2. On register page, click "I'm hiring" toggle
  3. Fill: Name = Priya Test, Email = client@test.com, Password = Client@12345
  4. Click Create account
  5. Open storage/logs/laravel.log → find the verification link → copy and open it
✅ Expected result: Account created, verification email logged. After clicking link → "Email verified" message. Can now login.
Mark:
AUTH-02 Register as Freelancer ▼
ℹ️ Do this in Browser B (incognito or different browser).
  1. Open localhost:5173/auth/register in Browser B
  2. Click "I'm a freelancer" toggle
  3. Fill: Name = Arjun Test, Email = freelancer@test.com, Password = Freelancer@12345
  4. Click Create account
  5. Find verification link in laravel.log → open in Browser B
✅ Expected: Freelancer account verified. Lands on freelancer dashboard.
AUTH-03 Login & Logout ▼
  1. Logout from Browser A (click "Sign out" in sidebar)
  2. Login again with client@test.com / Client@12345
  3. Verify you land on /client dashboard
  4. Logout → verify redirect to login page
✅ Expected: Login redirects to correct role dashboard. Logout clears session. Accessing /client without login redirects to /auth/login.
AUTH-04 Role guard — wrong role redirected ▼
  1. Login as client (Browser A)
  2. Manually navigate to localhost:5173/freelancer
  3. Check what happens
✅ Expected: Redirected to /403 Forbidden page. Cannot access other role's dashboard.
AUTH-05 Password reset flow ▼
  1. Go to /auth/forgot-password
  2. Enter client@test.com → click Send
  3. Find reset link in laravel.log → open it
  4. Enter new password Client@99999 → submit
  5. Login with new password
  6. Change password back to Client@12345 (repeat steps)
✅ Expected: Reset link works. New password login succeeds. Old password no longer works.
AUTH-06 Admin login ▼
  1. Open Browser C (or incognito) → localhost:5173/auth/login
  2. Login: admin@operalyn.com / Admin@123456
  3. Verify landing on /admin dashboard
  4. Check that all 9 nav items are visible (Dashboard, Users, Verifications, etc.)

Contracts & Milestones
5 test cases
CONT-01 Client adds milestones to contract ▼
  1. Browser A → Contracts → click the contract
  2. Click + Add milestone
  3. Add: Title = UI Design & Wireframes, Amount = 8000, Due date = 7 days from now
  4. Add: Title = Backend API Development, Amount = 10000
  5. Add: Title = Final Delivery & Testing, Amount = 4000
  6. Total should be ₹22,000 (matches contract total)
✅ Expected: 3 milestones created. Total allocated = ₹22,000. "₹0 unallocated" shown.
CONT-02 Freelancer submits delivery ▼
  1. Browser B → Contracts → click contract
  2. Find first milestone "UI Design" → click Submit work
  3. Write delivery note (20+ chars): "Completed all wireframes and UI designs as discussed. Files attached."
  4. Optionally attach a PDF file
  5. Click Submit delivery
✅ Expected: Milestone status changes to "Awaiting review". Client can now see delivery note and files.
CONT-03 Client requests revision ▼
  1. Browser A → contract detail → first milestone shows "Awaiting review"
  2. Click Request revision
  3. Enter notes: "Please add mobile responsive designs"
  4. Click Request revision
  5. Browser B → verify milestone now shows "Revision requested" with the notes
✅ Expected: Status → "Revision requested". Submit work button reappears for freelancer. Revision notes visible in delivery history.
CONT-04 Client approves milestone (Razorpay stub) ▼
⚠️ In dev, Razorpay keys are not set so approve falls back to direct approval. No payment modal will open — this is expected.
  1. Browser B → resubmit delivery for first milestone with revised note
  2. Browser A → milestone shows "Awaiting review" again
  3. Click Approve → confirm dialog
  4. Verify milestone status → Approved
  5. If Razorpay keys are set: Checkout modal should open → use test card 4111 1111 1111 1111
✅ Expected (no keys): Milestone marked Approved immediately. Expected (with keys): Razorpay checkout opens, after payment → milestone marked Paid.
CONT-05 Contract auto-completes when all milestones approved ▼
  1. Deliver + approve all 3 milestones (repeat CONT-02 + CONT-04 for milestones 2 and 3)
  2. After last milestone approved, check contract status
  3. "Leave a review ⭐" link should appear in contract header
✅ Expected: Contract status = "completed". Both parties see "Leave a review" link. Project status = "completed".

Real-time Chat
4 test cases
CHAT-01 Send and receive messages in real-time ▼
ℹ️ Chat server must be running: node server.js in E:\operalyn-chat
  1. Browser A → Contracts → contract → click 💬 Open chat
  2. Browser B → same (Messages → click conversation)
  3. Browser A: type "Hello Arjun, let's discuss the project" → press Enter
  4. Verify message appears in Browser B without refreshing
  5. Browser B: reply "Hi Priya, sounds great!"
  6. Verify reply appears in Browser A in real-time
✅ Expected: Messages appear in real-time on both sides. No page refresh needed. Messages persist after refresh.
CHAT-02 Typing indicator ▼
  1. Both browsers have the chat open
  2. Browser A: start typing (don't send)
  3. Check Browser B header — should show "typing…" animation
  4. Stop typing for 3 seconds
  5. Typing indicator should disappear in Browser B
✅ Expected: "typing…" appears while other party types. Disappears after 3 seconds of inactivity automatically.
CHAT-03 Online status indicator ▼
  1. Browser A has chat open — check for green dot next to Arjun's name
  2. Browser B: close the chat tab (or navigate away)
  3. Browser A: green dot should disappear (may take a few seconds)
  4. Browser B: reopen chat → green dot returns
✅ Expected: Green online dot appears when other party is in the chat. Disappears when they leave.
CHAT-04 Message history persists after refresh ▼
  1. Note the messages sent in CHAT-01
  2. Hard refresh Browser A (Ctrl+Shift+R)
  3. Navigate back to the chat
  4. All previous messages should be there
✅ Expected: All messages visible after refresh. Chat history loaded from Laravel DB, not socket memory.
